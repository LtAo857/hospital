# Fine-Tuning Plan

## Goal

Fine-tune a small Chinese-capable LLM for hospital registration NLU.
The model should convert user text into strict JSON:

```json
{
  "intent": "registration",
  "slots": {
    "symptom": "牙疼",
    "department": "口腔科",
    "doctorName": null,
    "date": "明天",
    "timePreference": null
  }
}
```

The model must not decide real medical facts, real schedules, or whether the
registration write can succeed. Those remain business-system responsibilities.

## Recommended Stack

- Base model: Qwen small model or another Chinese-capable open model.
- Method: SFT with LoRA or QLoRA.
- Training framework: LLaMA-Factory.
- Serving framework: vLLM for GPU deployment, or Ollama/llama.cpp for local demo.

## Data

Use `data/sft_samples.jsonl` as the interview demo format:

```json
{
  "instruction": "从用户输入中提取挂号意图和槽位，输出严格JSON。",
  "input": "明天牙疼，帮我挂个号",
  "output": "{\"intent\":\"registration\",\"slots\":{\"symptom\":\"牙疼\",\"department\":\"口腔科\",\"doctorName\":null,\"date\":\"明天\",\"timePreference\":null}}"
}
```

Data sources for a production version:

- historical agent conversations
- manually labeled bad cases
- weak labels generated by the current rule engine
- synthetic variants for dates, departments, symptoms, and doctor names

## Training Shape

1. Normalize labels into one schema.
2. Split train/eval data.
3. Run SFT with LoRA/QLoRA.
4. Evaluate intent accuracy, slot accuracy/F1, JSON valid rate, and unsupported
   intent recall.
5. Export LoRA adapter.
6. Serve `base model + adapter` through the inference service.

## Safety Boundary

The fine-tuned model output is advisory. The Java system must still:

- query real departments, doctors, schedules, and slots from MySQL
- validate login and patient card state
- enforce duplicate registration and daily limit rules
- require server-side confirmation before write operations
- fall back to rule parsing on timeout, low confidence, or invalid JSON

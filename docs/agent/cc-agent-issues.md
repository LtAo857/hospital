# 第三套 CC Agent 问题总结

## 背景

第三套架构用于和前两套做对照：

- 第一套：当前项目原有的规则编排式 `/agent/chat`
- 第二套：传统 ReAct 风格 `/agent/react/chat`
- 第三套：Claude Code / CC 风格 `/agent/cc/chat`

第三套要求和前两套隔离实现，不修改第一套代码，只在独立包和独立页面中演进。

## 当前入口

- 后端接口：`POST /agent/cc/chat`
- 后端代码：`patient-wx-api-mysql/src/main/java/com/example/hospital/patient/wx/api/agent/cc/`
- 小程序页面：`patient-wx/user/cc_chat/cc_chat.vue`

## 已落地能力

- 独立控制器、执行器、LLM 服务、工具注册表、Redis 会话记忆
- 独立前端页面，不影响第一套和第二套
- 独立会话前缀：`cc_agent_session:`
- 支持挂号主链路：科室 -> 诊室 -> 日期 -> 医生 -> 时段 -> 确认挂号

## 遇到的问题与修复

### 1. 小程序页面加载失败

现象：

- 第三套页面可以注册路由，但页面打不开或白屏

原因：

- `cc_chat.vue` 早期内容存在编码损坏
- 编译后生成的 `wxml` 出现标签和属性断裂
- 页面里还夹带了部分小程序兼容性风险写法

修复：

- 重写 `patient-wx/user/cc_chat/cc_chat.vue`
- 清理损坏文本
- 改成更保守的小程序写法
- 弹窗和卡片逻辑重新对齐

结果：

- 第三套页面可正常编译和加载

### 2. 用户回复医生名后重复返回医生列表

现象：

- 先返回“2026-04-12 可出诊的医生有：程淳美、赵六”
- 用户回复“赵六”
- 系统再次返回同一份医生列表，而不是继续查时段

原因：

- 旧逻辑只会识别“赵六医生”“赵六主任医师”这种带后缀写法
- 单独回复“赵六”不会写入 `doctorName`
- 状态机仍停留在“已知日期和诊室，未知医生”

修复：

- 从 `lastObservation.doctors` 中补做候选医生名匹配
- 用户短回复命中候选列表时，直接写入 `doctorName`
- 若切换医生，自动清空旧的 `doctorId/workPlanId/scheduleId/slot`

结果：

- 用户回复裸名字后，会继续查该医生时段，不再原地重复

### 3. 首轮一句话里直接带医生名，系统仍停在医生列表

现象：

- 用户直接说“帮我挂明天赵六的口腔科”
- 系统还是先回“可出诊的医生有……你想挂哪位医生”

原因：

- 修复第 2 个问题后，只覆盖了“候选列表之后的短回复”
- 首轮消息里的裸医生名仍未被识别

修复：

- 增加医生目录匹配逻辑
- 首轮消息如果包含系统已有医生名，也直接写入 `doctorName`

结果：

- “帮我挂明天赵六的口腔科”会继续走到时段查询

### 4. 无法理解“最早 / 最晚”

现象：

- 用户说“帮我挂明天赵六最早的口腔科”
- 系统只能把整串可挂时段返回给用户，不能自动选最早时段

原因：

- 旧逻辑只支持显式时段，如 `8点`、`08:00`
- `最早/最晚` 属于偏好，不是实体
- LLM 在当前架构里只负责选工具，不负责凭空生成具体 slot
- 查到 `slots` 后没有二次消费偏好的逻辑

修复：

- 在请求解析阶段增加 `slotPreference`：
  - `最早` -> `earliest`
  - `最晚` / `最后` -> `latest`
- 在拿到 `slots` 后自动选择最早或最晚可挂时段
- 自动写入 `workPlanId/scheduleId/slot/slotLabel`
- 然后进入确认挂号

结果：

- “帮我挂明天赵六最早的口腔科”会自动选择 `08:00`

### 5. 确认挂号阶段没有显示医生、日期、时段

现象：

- 系统进入确认态时，只显示“已经规划好提交挂号，请确认是否继续”
- 没有看到赵六、日期、时段等关键信息

原因：

- 后端确认文案是固定字符串
- 前端弹窗也写死成固定字符串
- 已经写进 payload 的 `doctorName/date/slot` 没被渲染出来

修复：

- 后端新增确认摘要拼装逻辑
- 摘要优先展示：
  - 日期
  - 诊室或科室
  - 医生
  - 时段
- 前端弹窗改为优先展示 `card.description`

结果：

- 确认态会显示类似：
  - `2026-04-12 口腔科 赵六 08:00`

## 当前第三套的能力边界

第三套当前已经能覆盖基础挂号链路，但仍然是“CC 风格实现”，不是完整 Claude Code 原生工作流复刻。

当前特点：

- LLM 只负责判断下一步调用哪个工具
- 真实业务数据完全来自后端工具查询
- 编排层负责：
  - 会话记忆
  - 状态推进
  - 写操作确认
  - 对模型决策做守卫

这意味着第三套已经具备：

- 可运行
- 可对比
- 可继续演进

但还没有做到：

- 更复杂的任务计划树
- 多步骤可视化计划管理
- 更丰富的偏好求解
- 更通用的长链路自治

## 当前结论

第三套已经从“独立页面 + 独立后端骨架”推进到“可实际跑通挂号主链路”的阶段。

本轮最重要的收获不是单点修 bug，而是明确了第三套在当前项目中的设计边界：

- 它不是第一套的替换品
- 它不是第二套的简单改名版本
- 它是一个独立的、偏计划模式的对照架构
- 现阶段重点是可运行、可解释、可对比、可继续扩展

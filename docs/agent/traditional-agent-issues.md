# 第二套传统 Agent 问题复盘与解决方案

## 背景

第二套架构是独立于第一套规则编排 Agent 的传统 ReAct 风格实现，后端入口为：

- 接口：`POST /agent/react/chat`
- 控制器：`patient-wx-api-mysql/.../agent/react/controller/TraditionalAgentChatController.java`
- 小程序页面：`patient-wx/user/react_chat/react_chat.vue`

这套架构的目标是保留“理解问题 -> 选工具 -> 观察结果 -> 再决策”的传统 Agent 形态，同时不改动第一套代码。

## 本轮暴露的主要问题

### 1. LLM 会提前收口，并输出未经过工具验证的结论

现象：

- 用户说“帮我挂明天的骨科”。
- 系统直接回复“某医生无号源”，甚至出现并未通过工具查到的医生名。

根因：

- LLM 被赋予了过强的最终回答权。
- `finish/finalAnswer` 在没有真实 observation 的情况下也会被采用。
- 模型生成的 `toolInput` 曾经可以直接影响医生、诊室等关键实体。

解决方案：

- 增加 `finish` 守卫，没有真实工具调用前禁止直接结束。
- 最终回复优先从最后一次真实 tool observation 生成，不再直接相信 LLM 的 `finalAnswer`。
- 将 LLM 的职责收缩为“决定下一步调用哪个工具”，不再信任其凭空补出的实体参数。
- 对工具选择增加前置条件校验，如果模型越级选择工具，会强制回退到更早的查询步骤。

效果：

- 第二套不会再仅凭模型猜测就断言“无号源”。
- 回复必须基于真实科室、诊室、医生、排班、时段数据。

### 2. 查到真实候选项后，走到最大步数才结束，最后吐英文摘要

现象：

- 用户问“帮我挂明天的骨科”。
- 工具已经查到了 `subDepartments`，但系统最后回复 `TraditionalAgent 已完成多步查询，当前结果是: Listed clinic rooms`。

根因：

- 执行器没有在“已经拿到足够回复用户的 observation”时及时结束。
- 循环跑到 `maxSteps` 后退化成内部 summary 输出。

解决方案：

- 增加 `shouldReplyAfterTool()`。
- 当 observation 已经足够支撑用户下一步选择时，直接结束本轮并生成中文 grounded reply。
- 即使走到 `maxSteps`，也优先使用最后一次真实 observation 生成中文结果，而不是输出英文内部 summary。

效果：

- “骨科下的诊室有：骨科三诊。你可以继续告诉我想查哪个诊室。”
- 不再出现 `Listed clinic rooms` 这类内部英文摘要直接暴露给用户。

### 3. 只有一个候选诊室或一个候选医生时，仍然要求用户再选一次

现象：

- 第二套会先回复“骨科下的诊室有：骨科三诊。你可以继续告诉我想查哪个诊室。”
- 用户回复“骨科三诊”后，还可能继续出现类似“2026-04-12 可出诊的医生有：王五。你可以继续告诉我想挂哪位医生。”

根因：

- 旧逻辑把“有候选项”统一视为分叉节点。
- 即使候选列表长度只有 1，也没有自动命中并继续向下钻取。

解决方案：

- 当 `subDepartments` 结果只有 1 个时，自动写入 `deptSubId/deptSubName`。
- 当 `doctors` 结果只有 1 个时，自动写入 `doctorId/doctorName`。
- 在停点判定中，只有“确实存在多个候选项”时才停下来等用户补充。

效果：

- “帮我挂明天的骨科”可以从唯一诊室自动推进到唯一医生，再继续查真实时段。
- 减少假选择和机械式追问。

### 4. 用户已经明确给了日期，系统仍停在日期可选列表

现象：

- 用户说“帮我挂明天的骨科”。
- 第二套有时会回复“我查到接下来可挂号的日期有：2026-04-12。你可以继续指定其中一天。”

根因：

- `list_register_dates` 只返回日期列表，没有标记“用户当前输入的日期已经命中”。
- 执行器把所有 `dates` observation 都当作需要向用户展示的中间候选。

解决方案：

- 在 `list_register_dates` 中增加 `matchedDate`。
- 如果用户已经明确指定日期并命中真实可挂号日期，则继续往下查医生，而不是停在日期列表。
- 只有在日期不明确或存在多个可选日期时，才把日期列表回给用户。

效果：

- “明天”会被解析为绝对日期，例如 `2026-04-12`，并在命中后继续下钻。

### 5. 用户选择具体时段后，系统重复返回时段列表

现象：

- 第二套回复：“王五在 2026-04-12 还有可挂时段：08:00（余3）……”
- 用户回复：“8点的”
- 系统又把整段时段列表重复返回一遍，而不是进入挂号确认。

根因：

- 用户文本中的“8点的 / 08:00”没有被解析成具体 `slot/workPlanId/scheduleId`。
- memory 中虽然已有 `doctorId + date`，但没有具体时段，因此 fallback 决策仍然继续走 `list_schedule_slots`。

解决方案：

- 新增时段解析逻辑，支持：
  - `08:00`
  - `8点`
  - `8点半`
  - `上午8点`
  - `下午2点30`
- 从上一轮真实 `lastObservation.slots` 中匹配对应时段。
- 命中后将 `slot/workPlanId/scheduleId/amount` 写入 memory。
- 增加 `slotSelectionPendingConfirmation` 标记。
- 一旦具体时段已经被选中，强制将下一步改写为 `create_registration`，进入写操作确认守卫，而不是再查一轮 slots。

效果：

- “8点的”不应再重复时段列表。
- 正确行为应转为“已为你选中 2026-04-12 08:00，确认后将提交挂号”一类确认态回复。

### 6. 编译通过但运行效果没有变化

现象：

- 代码已经修改并本地 `mvn -q -DskipTests compile` 通过。
- 但实际接口返回仍然是旧行为。

根因：

- 正在运行的 `patient-wx-api-mysql` 进程没有重启，线上实际返回仍来自旧 class。

解决方案：

- 文档修改、代码修改后，需要重启 `patient-wx-api-mysql` 服务。
- 尤其是第二套 Agent 的执行器、工具注册表、fallback 逻辑这类改动，只有新进程加载后才能生效。

效果：

- 编译结果与实际接口行为保持一致。

## 当前第二套架构的设计结论

第二套传统 Agent 不适合让 LLM 直接承担“最终事实判断器”的职责，更稳妥的边界应当是：

- LLM 负责选下一步工具
- 真实业务工具负责产出事实
- 编排层负责做状态守卫、前置条件校验、单候选自动推进、写操作确认
- 最终回复尽量从真实 observation 生成

这也是本轮改造的核心方向。

## 建议的后续优化

- 为“单候选自动推进”补充前端卡片态展示，让用户知道系统做了什么自动选择。
- 对时段选择补充卡片按钮，避免完全依赖自然语言解析。
- 在 tool log 中区分“模型原始决策”和“编排层改写后的决策”，方便排障。
- 为第二套架构补充接口级回放测试，覆盖：
  - “帮我挂明天的骨科”
  - “骨科三诊”
  - “王五”
  - “8点的”
  - “确认挂号”

# 第一套 Agent 问题复盘与解决方案

## 背景

第一套架构是当前项目已经落地的患者侧挂号助手主链路，核心特点是：

- 入口接口：`POST /agent/chat`
- 编排核心：`AgentOrchestratorService`
- 基础决策：`NoModelAgentEngine`
- 可选意图识别：`DashScopeAgentService`
- 总体模式：规则编排为主，LLM 只做结构化意图识别与弱辅助

这套架构的目标不是做一个“完全自主规划”的通用 Agent，而是做一个面向挂号业务的稳定编排层。

## 第一套在实现过程中遇到的主要问题

### 1. 模型容易把日期识别错，覆盖用户原话

现象：

- 用户说“明天”“后天”或给出具体日期。
- 如果直接采用模型返回的 payload，可能出现模型把日期理解错，最终查询到错误日期的号源。

根因：

- 纯靠模型做意图识别时，日期这种相对时间很容易被误解析。
- 模型输出一旦直接进入业务流程，就会把错误日期带到后续查询。

解决方案：

- 在编排层单独解析用户原话中的日期提示。
- 将 `今天 / 明天 / 后天 / yyyy-MM-dd / x月x日` 统一转成绝对日期。
- 用户原话中的日期优先级高于模型返回值。

当前结论：

- 第一套已经把“用户原话日期优先”作为硬规则，而不是模型建议。

### 2. 模型可能在多轮中把流程带偏，重置到起点

现象：

- 用户明明已经在“选日期”或“选医生”的中间阶段。
- 模型却可能再次返回 `start_registration` 或 `view_departments`，导致流程回退。

根因：

- LLM 更擅长理解单轮语义，不天然擅长严格遵守业务状态机。
- 多轮挂号流程本质是有状态的，需要由编排层而不是模型自己维持阶段。

解决方案：

- 在 memory 中引入 `stage`。
- 通过 `resolveActionByStage`、`NoModelAgentEngine.resolveAction()` 等规则，优先延续当前流程阶段。
- 即使启用模型，也要求它参考 `memory.stage`，不能随意重开流程。

当前结论：

- 第一套本质是“状态机 + 工具编排”，不是让模型自由规划。

### 3. 症状词无法直接映射到科室，用户体验不自然

现象：

- 用户说“牙痛”“口腔难受”“智齿疼”。
- 如果系统只识别明确科室名，就无法顺畅进入挂号流程。

根因：

- 医疗挂号场景里，很多用户先表达症状，不会直接说标准科室名。
- 如果没有业务规则补全，系统只能机械追问“请先告诉我科室”。

解决方案：

- 在编排层增加症状词到科室的映射规则。
- 当前已补的典型映射是：`口腔 / 牙痛 / 牙疼 / 牙龈 / 智齿 / 牙周` -> `口腔科`。

当前结论：

- 第一套已经不完全依赖模型理解症状，而是通过业务规则做显式补全。

### 4. 医生没有价格记录时，被错误地从查询结果中过滤

现象：

- 某些医生明明有排班、有号源。
- 但因为医生价格字段缺失，自动挂号或医生列表中会看不到这位医生。

根因：

- 旧逻辑里把价格信息错误当成了可挂号的前置条件。
- 这属于“展示字段”和“业务可挂号条件”混淆。

解决方案：

- 调整工具约束：医生价格不是号源查询的前置条件。
- 没有价格记录时不再过滤医生，价格按 `0.00` 或空值兜底展示。

当前结论：

- 第一套已把“医生是否可展示 / 可挂号”和“是否配置价格”解耦。

### 5. 自动挂号会选到当天已经过期的时段

现象：

- 用户说“今天挂号”。
- 如果系统只按“最早 slot 编号”去选，有可能挑中当天已经过去的时间段。

根因：

- 排班数据只说明“这个时段存在”，不代表“这个时段此刻仍可挂”。
- 自动挂号需要在业务时间维度上进一步过滤。

解决方案：

- 在自动挂号候选筛选里加入“当天时段是否已过期”的判断。
- 如果当天仍有排班但所有剩余时段都已过，则不再自动下单。
- 明确提示“今天的剩余可挂时段已经结束了”。

当前结论：

- 第一套已将“存在排班”与“当前时刻仍可挂”区分处理。

### 6. 只靠模型做挂号决策，风险太高

现象：

- 如果让模型自己决定“能不能挂、挂哪个、是否提交”，容易出现误判。
- 这类问题在挂号场景里属于高风险错误，因为会直接影响真实订单提交。

根因：

- LLM 不具备天然的强一致性和严格约束能力。
- 挂号涉及实名、就诊卡、重复挂号、每日上限、号源竞争等硬条件，不能靠语言模型猜。

解决方案：

- 第一套从架构上将模型能力限制为“意图识别”。
- 真正的业务判断统一落到工具与编排层：
  - `checkRegistrationCondition`
  - `createRegistrationOrder`
  - 会话阶段控制
  - 用户确认守卫

当前结论：

- 第一套不支持更复杂的自由规划，不是能力缺失，而是刻意收缩边界后的稳定性选择。

### 7. 写操作如果没有确认守卫，业务风险不可接受

现象：

- 挂号不是普通查询，而是真实写操作。
- 如果系统在没有显式确认的情况下直接调用挂号接口，会带来误下单风险。

根因：

- 用户表达“帮我挂”不等于“立即提交订单”。
- 中间还可能缺实名、缺就诊卡、重复挂号、限号冲突，或者用户只是想先看号源。

解决方案：

- 第一套将挂号拆成两段：
  1. 查询并生成候选挂号参数
  2. `checkRegistrationCondition` 校验通过后进入确认态
- 只有用户明确确认后，才调用 `createRegistrationOrder`

当前结论：

- 第一套对写操作采用强确认，不允许模型或规则直接越过确认提交。

### 8. 自动挂号找不到候选时，不能只说“失败”

现象：

- 自动挂号未找到合适候选时，如果只返回“没有号源”或“失败”，用户无法继续操作。

根因：

- 自动挂号只是“快捷路径”，失败不代表整个挂号流程失败。
- 用户仍然可能通过手动选择诊室、日期、医生完成挂号。

解决方案：

- 自动挂号失败时，第一套会把流程切回手动选择阶段。
- 例如：
  - `choose_sub_department`
  - `choose_date`
  - `choose_doctor`
  - `choose_slot`
- 同时返回卡片和下一步引导，而不是直接结束会话。

当前结论：

- 第一套更像“快捷自动化 + 可回退手动流程”的混合编排，而不是单路径自动挂号。

## 为什么第一套没有做成传统通用 Agent

第一套遇到的这些问题，本质都指向同一件事：

- 挂号不是开放问答，而是强状态、强约束、强确认的业务流程。

因此第一套最终选择的是：

- LLM 只做意图识别
- 规则层负责阶段控制和关键字段修正
- 工具层负责真实业务查询
- 写操作必须经过条件校验和用户确认

这也是为什么第一套不支持“更复杂的自由规划”：

- 复杂规划会放大模型的不确定性
- 多轮业务状态更容易被带偏
- 对真实挂号写操作不够安全

## 当前第一套的设计结论

第一套已经沉淀出的稳定原则是：

- 用户原话优先于模型猜测
- 状态机优先于自由规划
- 工具事实优先于文本推理
- 写操作确认优先于自动化便捷

这套原则更适合当前项目的真实挂号业务。

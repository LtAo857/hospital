# 医院挂号系统 + AI 挂号助手项目面经

> 基于当前仓库整理，默认聚焦 MySQL 链路：`hospital-api-mysql`、`patient-wx-api-mysql`、`hospital-vue`、`patient-wx`。
>
> 适用场景：简历项目介绍、Java 后端面试、全栈联调面试、AI 应用开发面试、项目复盘。

---

## 1. 项目一句话介绍

这是一个面向患者、医生和医院管理者的智慧医疗挂号系统，包含：

- 患者端小程序：挂号、消息、评价、收藏、电子处方、视频问诊、AI 挂号助手
- 管理端 MIS：医生/科室/排班/挂号/收藏/评价/电子处方/视频问诊管理
- 后端服务：患者侧 API + 管理侧 API
- AI 能力：在患者端新增任务型挂号助手，不只是聊天，而是“查询 -> 条件校验 -> 确认 -> 挂号”的可执行工作流

如果面试官只给你 20 秒，你可以这样说：

> 我做的是一个医院挂号系统，前端分患者小程序和医生/管理员 MIS，后端分两套 Spring Boot 服务。核心是把科室、诊室、医生、排班、号源、挂号、处方、评价、收藏、视频问诊这些医疗业务做线上化。另外我还结合现有挂号流程做了一个 AI 挂号助手，让用户可以通过对话完成挂号查询和确认提交。

## 3. 3 分钟项目介绍话术

### 3.1 面试可直接说的版本

> 我这个项目是一个智慧医疗挂号系统，分患者小程序端和医生/管理员 MIS 管理端。技术上后端是 Spring Boot + MyBatis + Redis + Quartz + WebSocket，前端患者侧是 uni-app，管理端是 Vue3 + Element Plus。  
> 项目核心业务是把医院线下的“科室 -> 诊室 -> 医生 -> 排班 -> 时段 -> 挂号”链路做成线上化。患者可以在小程序按未来几天号源进行挂号，挂号成功后系统会推送消息提醒；医生端和患者端还支持视频问诊；挂号之后还能查看电子处方、提交评价、收藏医生，形成一个完整闭环。  
> 我重点做和关注的部分有三块：第一是挂号并发控制，因为同一个时段可能有多人同时抢号，所以我在挂号提交链路里结合 Redis 做了并发控制，避免号源超卖；第二是视频问诊和消息通知，后台用 WebSocket 推送业务状态，前端结合 TRTC 做音视频通话；第三是 AI 挂号助手，我不是单纯做聊天，而是把现有挂号能力封装成工具，让 Agent 通过多轮对话完成科室、日期、医生、时段的查询与确认，最后再调用挂号接口真正落单。

可以拆成 5 个点：

1. 设计并实现挂号主链路：科室、诊室、日期、医生、时段、挂号记录
2. 做了消息提醒闭环：挂号成功消息 + 每日就诊提醒
3. 做了医生评价、收藏、电子处方等后置功能
4. 做了视频问诊：WebSocket + TRTC
5. 做了 AI 挂号助手：规则编排 / ReAct / CC 三套对照架构

## 4. 核心业务链路，面试一定要会讲

---

### 4.1 门诊挂号主链路

这是整个项目最核心的业务链路。

#### 业务流程

1. 患者进入科室列表
2. 选择诊室
3. 查看近 7 天可挂号日期
4. 选择当天出诊医生
5. 选择医生可挂号时段
6. 检查挂号条件
7. 创建挂号记录
8. 发送挂号成功消息

#### 关键代码落点

- 小程序 API 地址定义：`patient-wx/main.js`
- 挂号服务实现：`patient-wx-api-mysql/src/main/java/com/example/hospital/patient/wx/api/service/impl/RegistrationServiceImpl.java`
- 近 7 天可挂号日期：`searchCanRegisterInDateRange()`
- 当天医生列表：`searchDeptSubDoctorPlanInDay()`
- 时段列表：`searchDoctorWorkPlanSchedule()`
- 条件校验：`checkRegisterCondition()`
- 创建挂号：`registerMedicalAppointment()`

#### 面试可讲重点

- 近 7 天日期不是死数据，而是根据医生工作计划动态返回
- 条件校验会限制重复挂号和当日挂号次数
- 真正的挂号写操作单独在 `registerMedicalAppointment()` 中执行
- 挂号成功后会同步发消息，不是只写数据库

#### 可包装亮点

> 这个链路不是简单 CRUD，而是强业务约束的流程型接口，我把查询、条件校验、写操作拆开，便于前端逐步推进，也为 AI 助手的工具化封装提供了基础。

### 4.2 挂号并发控制：防止号源超卖

这是最适合讲技术深度的点。

#### 当前实现思路

在 `RegistrationServiceImpl.registerMedicalAppointment()` 中，系统会：

1. 先根据 `scheduleId` 拼出 Redis key，例如 `doctor_schedule_{scheduleId}`
2. 判断 Redis 是否有该时段缓存
3. 用 Redis `watch -> multi -> exec` 做乐观锁事务
4. 只有 `num < maximum` 才允许已挂号人数加 1
5. Redis 成功后再写 MySQL 挂号记录
6. 同步更新工作计划表和工作计划时段表中的已挂号数量
7. 成功后发送挂号成功消息

#### 为什么这点值得讲

因为医疗挂号和电商库存类似，面试官会天然追问你怎么解决并发超卖问题。

#### 标准回答模板

> 我们不是直接查数据库再更新，而是先在 Redis 中维护时段维度的号源数据。提交挂号时通过 `watch + multi + exec` 做乐观锁控制，只有缓存里的 `num < maximum` 时才允许占用一个号源。Redis 提交成功后再落 MySQL 挂号记录和统计字段，这样能降低高并发下同一时段的超卖风险。

#### 面试官继续追问：那 Redis 成功、DB 失败怎么办？

你可以这样答：

> 这是典型的缓存与数据库双写一致性问题。当前代码里已经有补偿思路的痕迹，比如异常时回滚 Redis 计数、支付过期恢复号源等，只是本地演示版支付链路做了简化。真正线上化的话，我会把这部分补成事务补偿、延迟对账、幂等保护，而不是只依赖单次请求成功。

#### 可主动说的工程折中

- 当前版本支付链路做了简化，挂号单直接写成成功状态，便于本地联调
- 这让挂号主链路更容易跑通，但也意味着支付补偿、超时恢复、最终一致性是后续可继续增强的方向

这样说不会显得你项目有问题，反而会显得你理解系统演进。

### 4.3 视频问诊链路

这是项目里另一个高价值亮点。

#### 业务流程

1. 医生在后台上线视频问诊
2. 后台可控制是否开放视频问诊号源
3. 患者进入视频问诊准备页，提交资料
4. 系统分配房间 / 查询房间信息
5. 医患双方获取 TRTC 鉴权并进入同一房间
6. 后台通过 WebSocket 推送开始问诊、结束问诊、刷新状态

#### 关键代码落点

- 管理端页面：`hospital-vue/src/views/video_diagnose.vue`
- 管理端 WebSocket：`hospital-api-mysql/src/main/java/com/example/hospital/api/socket/WebSocketService.java`
- 管理端视频问诊接口：`hospital-api-mysql/src/main/java/com/example/hospital/api/controller/VideoDiagnoseController.java`
- 患者端视频问诊接口：`patient-wx-api-mysql/src/main/java/com/example/hospital/patient/wx/api/controller/VideoDiagnoseController.java`
- 患者端视频问诊服务：`patient-wx-api-mysql/src/main/java/com/example/hospital/patient/wx/api/service/impl/VideoDiagnoseServiceImpl.java`
- 患者端页面：`patient-wx/video_diagnose/video_diagnose/video_diagnose.vue`

#### 面试要点

- WebSocket 负责业务状态推送，不负责音视频流
- TRTC 负责音视频传输，不负责业务状态同步
- 二者结合才完整：一个推状态，一个传媒体流

#### 可直接背的回答

> 视频问诊这块我把“实时状态”和“音视频传输”拆开处理。业务状态比如开始问诊、结束问诊、排队刷新，用 WebSocket 推给医生端；真正的音视频通信通过 TRTC 完成。这样职责更清晰，也更容易排查问题。

### 4.4 消息提醒链路

#### 当前已落地的两种提醒

1. 挂号成功消息：挂号成功后立即发送
2. 每日就诊提醒：Quartz 每天早上固定时间执行

#### 关键代码落点

- 挂号成功后发消息：`RegistrationServiceImpl.registerMedicalAppointment()`
- Quartz 配置：`patient-wx-api-mysql/src/main/java/com/example/hospital/patient/wx/api/config/QuartzConfig.java`
- 就诊提醒任务：`patient-wx-api-mysql/src/main/java/com/example/hospital/patient/wx/api/job/RegistrationReminderJob.java`
- 消息服务：`patient-wx-api-mysql/src/main/java/com/example/hospital/patient/wx/api/service/impl/MessageServiceImpl.java`

#### Quartz 当前调度规则

- 每天早上 8 点执行一次就诊提醒

#### 面试可讲点

- 用 Quartz 而不是前端轮询，是因为提醒属于服务端业务任务
- Quartz 适合承接后续的支付超时、复诊提醒等任务扩展
- 就诊提醒不是定时发死文案，而是查当天挂号患者后逐条生成消息

#### 面试官继续问：会不会重复发？

你可以答：

> 当前版本以单机调度和业务链路跑通为主，后续如果做成多实例部署，我会补任务幂等和消息去重，比如按 `userId + registrationId + date + type` 建唯一标识，防止重复提醒。

### 4.5 电子处方链路

#### 业务价值

挂号不是终点，医生问诊后可以开处方，患者可按挂号单查看处方详情，这是一个完整医疗闭环。

#### 关键代码落点

- 管理端处方服务：`hospital-api-mysql/src/main/java/com/example/hospital/api/service/impl/DoctorPrescriptionServiceImpl.java`
- 管理端处方页面：`hospital-vue/src/views/doctor_prescription.vue`
- 患者端处方接口：`patient-wx-api-mysql/src/main/java/com/example/hospital/patient/wx/api/controller/PrescriptionController.java`
- 患者端处方服务：`patient-wx-api-mysql/src/main/java/com/example/hospital/patient/wx/api/service/impl/PrescriptionServiceImpl.java`
- 患者端页面：`patient-wx/registration/prescription/prescription.vue`

#### 当前实现特点

- 医生端按挂号单开立处方
- 处方药品列表以 JSON 数组形式存储
- 后端保存前会校验 JSON 格式是否合法
- 患者端按挂号单查询并解析成 `rpList`

#### 面试可讲点

- 为什么处方内容用 JSON 存：列表结构灵活，前后端传输方便

- 还做了权限控制：不能操作其他医生的挂号单

#### 一句总结

> 处方模块体现的是“结构化业务数据 + 权限校验 + 双端联动展示”的能力。

## 5. AI 挂号助手：这个项目最差异化的部分

这是你和普通 Java 项目拉开差距的重点。

---

### 5.1 AI 助手不是普通聊天框

很多项目写“接了个大模型”，面试时一问就露馅，因为本质只是把用户输入转发给模型。

这个项目不是这样。

当前患者端 AI 助手是真正的任务型 Agent：

- 前端页面会展示步骤流转 `steps`
- 会展示可执行卡片 `cards`
- 会展示工具执行日志 `toolLogs`
- 会在真正写操作前做确认
- 后端会持久化对话记忆
- 工具层复用了真实业务服务，而不是造假数据

#### 关键代码落点

- 前端页面：`patient-wx/agent/chat/chat.vue`
- 接口入口：`patient-wx-api-mysql/src/main/java/com/example/hospital/patient/wx/api/agent/controller/AgentChatController.java`
- 主编排器：`patient-wx-api-mysql/src/main/java/com/example/hospital/patient/wx/api/agent/service/AgentOrchestratorService.java`
- 会话记忆：`patient-wx-api-mysql/src/main/java/com/example/hospital/patient/wx/api/agent/memory/RedisAgentConversationMemoryService.java`
- 工具定义说明：`docs/agent/tool-spec.md`

---

### 5.2 一期架构：规则编排 + 可选 LLM 决策

#### 当前思路

一期不是完全依赖大模型，而是：

1. 本地规则引擎先做基础意图识别
2. 如果开了 LLM，就让模型返回结构化意图
3. 编排层再合并模型结果和本地规则
4. 编排层强制做关键字段修正
5. 只有满足业务前置条件才会真正推进到下一步

#### 关键工程思想

- LLM 不是最终事实来源
- 模型不能绕过工具直接写业务
- 读操作可以自动执行
- 写操作必须确认

#### 适合面试时说的一句话

> 我们不是让模型直接决定挂号结果，而是把模型限制在“理解意图、推荐下一步”的角色上，真实事实始终来自工具查询，真正的写操作还要经过编排层确认和业务校验。

---

### 5.3 一期具体做了哪些 Guardrail

这部分越具体越能体现你真的懂 AI 工程。

#### 当前已落地的约束

1. **症状词映射科室**  
   比如“牙痛”“口腔难受”会优先补全到 `口腔科`

2. **用户原话中的日期优先**  
   如果用户说了“今天 / 明天 / 后天”或明确日期，优先按用户原话解析，不让模型覆盖真实日期

3. **当天已过时段不自动选择**  
   自动挂号时会过滤掉当天已经过期的时段

4. **医生价格不是硬前置条件**  
   即使医生暂时没配置价格，也不会把医生整条过滤掉

5. **写操作必须确认**  
   `create_registration` 之前必须走确认态

6. **挂号前再次检查条件**  
   即使前面查询都通过，最终写入前还是会再做 `checkRegistrationCondition`

#### 这套机制解决什么问题

- 幻觉式回答
- 模型越权下单
- 日期误识别
- 当天过期号源误选
- 用户意图已明确但流程没推进

---

### 5.4 多轮会话记忆怎么做

#### 当前实现

- 使用 Redis 保存会话 memory
- key 前缀：`agent_session:`
- 保存结构：`Map<String, Object>`
- 会保存 `stage`、`deptName`、`deptSubName`、`date`、`doctorName`、`pendingOrder` 等状态
- TTL 由配置控制，当前代码中为 30 分钟

#### 为什么不用前端自己记

因为：

- 前端刷新会丢状态
- 服务端更适合承接多轮对话上下文
- 后续可以横向扩展，不把状态绑死在单个客户端上

#### 标准回答模板

> 多轮对话本质上也是有状态工作流，所以我把会话状态放到了 Redis，而不是只放前端。这样用户在一次会话里已经选择过的科室、日期、医生、待确认订单都能复用，Agent 每轮只需要补足缺失参数。

---

### 5.5 为什么要做三套 Agent 架构对照

仓库里不是只有一套 Agent，而是三套：

1. **一期**：规则编排式 `/agent/chat`
2. **二期**：传统 ReAct 风格 `/agent/react/chat`
3. **三期**：CC 风格 `/agent/cc/chat`

#### 为什么这点很有价值

因为这不是简单“换个接口名”，而是在比较不同 AI 应用架构的边界：

- 谁来做最终事实判断
- 谁负责选工具
- 谁负责流程推进
- 谁来守卫写操作
- 多轮对话的状态怎么存

#### 面试可讲结论

> 我们实际踩坑后发现，在医疗挂号这类强约束业务里，不适合让 LLM 直接做最终事实判断器。更稳定的做法是：LLM 负责选下一步工具，真实工具负责产出事实，编排层负责状态守卫和写操作确认。

这个回答很有“AI 应用工程化”的味道。

---

### 5.6 二期 ReAct 架构踩过什么坑

这部分建议你面试一定要讲，因为“踩坑 -> 修复 -> 总结”最能体现实战深度。

#### 典型问题

1. 模型会提前收口，输出没有被工具验证过的结论
2. 已经查到候选诊室/医生后，还继续跑到最大步数，最后吐英文内部摘要
3. 只有一个候选诊室或医生时，还让用户再选一次
4. 用户已经明确给了日期，还停在日期列表不往下走
5. 用户选了时段后，又重复把时段列表返回一遍

#### 修复方向

- LLM 只选工具，不直接给最终结论
- 最终回复尽量从真实 observation 生成
- 单候选自动推进
- 对明确日期、明确医生、明确时段做命中解析
- 一旦命中具体时段，就进入确认态，而不是继续重复查

#### 面试价值

这类问题说明你不是停留在“接个模型 API”，而是在做真正的 Agent 行为调优。

### 5.7 三期 CC 风格架构踩过什么坑

#### 典型问题

1. 小程序页面加载失败，属于前端页面结构与编码问题
2. 用户只回医生名字，系统重复返回医生列表
3. 首轮一句话已经带了医生名字，系统仍然停在医生列表
4. 不理解“最早 / 最晚”这种偏好词
5. 进入确认态时没把医生、日期、时段摘要展示出来

#### 修复方向

- 通过候选医生列表做裸名字匹配
- 让首轮医生名也能被识别并直达时段查询
- 在请求解析阶段加入 `slotPreference`
- 自动把“最早 / 最晚”映射成时段选择策略
- 确认态优先展示具体摘要，而不是空泛地说“是否继续”

#### 很适合面试时讲的工程结论

> 用户并不总是按我们预期的按钮流来操作，所以 AI 工作流设计必须兼容“自由文本 + 候选列表 + 状态记忆”三种输入形态，否则用户会觉得它很笨。

## 7. 项目难点与标准回答

这一节建议重点记住，因为面试官最容易围绕这些问。

---

### 难点 1：挂号并发与库存超卖

**问题**：同一医生同一时段有多个用户同时抢号，怎么防止超卖？

**回答**：

> 我们把时段号源信息缓存到 Redis，挂号时不直接查库扣减，而是先对 `doctor_schedule_{scheduleId}` 做 `watch + multi + exec` 乐观锁事务，只有当前已挂号人数小于最大人数时才允许加 1。抢占成功后再写挂号记录和统计字段，这样能把并发竞争尽量前置到 Redis 处理。

**延伸点**：

- 为什么不用数据库行锁直接做？
- Redis 成功、数据库失败如何补偿？
- 如果 Redis key 不存在怎么办？

---

### 难点 2：缓存和数据库一致性

**问题**：Redis 和 MySQL 双写怎么保证一致？

**回答**：

> 当前版本的重点是先把挂号主链路跑通，所以做法是 Redis 抢占成功后再落数据库。这个方案能较好控制并发，但也会带来典型的一致性问题。我的思路是把这部分分成两个层次：一是请求内异常尽量补偿，比如异常时回滚 Redis；二是请求外再用延迟任务或对账任务修正异常状态。也就是说，这块不能只靠本地事务，而要用补偿思维看。

---

### 难点 3：视频问诊中的双通道协作

**问题**：为什么要同时用 WebSocket 和 TRTC？

**回答**：

> 因为它们解决的是两类问题。WebSocket 解决业务状态同步，比如开始问诊、结束问诊、刷新队列；TRTC 解决音视频媒体流传输。如果只用 TRTC，业务状态不好管；如果只用 WebSocket，就没有实时音视频。两者组合才完整。

---

### 难点 4：AI 助手怎么避免胡说八道

**问题**：大模型最容易幻觉，你怎么保证它不乱说？

**回答**：

> 我们在设计上就不让模型直接当事实来源。模型最多负责判断下一步该调哪个工具，真实的科室、医生、日期、时段都必须来自工具查询。编排层还会对模型输出做二次修正，比如用户原话日期优先、症状词映射科室、当天已过时段过滤。最终写操作还必须经过确认和业务校验，所以不会让模型直接改数据库。

---

### 难点 5：多轮会话状态怎么维护

**问题**：如果用户说“帮我挂明天口腔科的号”，下一句又说“最早的”，系统怎么记住上下文？

**回答**：

> 我们把对话状态放在 Redis，而不是只放前端。每次请求会读出当前 session 的 memory，里面有 stage、科室、诊室、日期、医生、待确认订单等状态，编排层会优先补齐缺失字段。这样用户第二轮只说“最早的”，系统也知道是在当前医生和当前日期的候选时段里继续筛选。

---

### 难点 6：自然语言不是按钮流，怎么推进任务

**问题**：用户不会严格按“点按钮 -> 下一步 -> 点按钮”来操作，你怎么处理？

**回答**：

> 我们把输入分成三类：自由文本、候选卡片点击、上下文记忆。比如用户可能直接说“帮我挂明天赵六最早的口腔科”，这时候不能要求他还要一步一步再选日期、医生、时段，而是要通过文本解析命中医生、日期和偏好，再借助工具查真实候选并自动推进到确认态。

---

## 8. 面试官高频项目追问 30 题

下面这 30 题很适合你反复练。

---

### 8.1 项目介绍类

#### 1）你的项目解决了什么问题？

解决医院挂号线上化问题，同时把挂号、处方、评价、收藏、视频问诊和消息提醒做成闭环，并在患者端加入 AI 挂号助手提升交互效率。

#### 2）这个项目的核心链路是什么？

科室 -> 诊室 -> 日期 -> 医生 -> 时段 -> 挂号 -> 消息 -> 处方 / 评价 / 收藏。

#### 3）你在这个项目里最有技术含量的部分是什么？

挂号并发控制、视频问诊实时状态同步、AI 挂号助手的工具编排和 Guardrail。

#### 4）为什么说这个项目不是简单 CRUD？

因为挂号是强业务流程，不只是增删改查，还涉及排班、号源、限制条件、消息闭环、并发控制、写操作确认。

---

### 8.2 挂号链路类

#### 5）挂号前会做哪些校验？

- 当天挂号次数限制
- 是否已挂过该诊室
- 是否有可用时段
- AI 助手场景下还会校验登录态与就诊卡

#### 6）为什么把条件校验和真正提交拆开？

因为前者可以在选择页面提前反馈，后者才是真正写操作。这样既利于前端交互，也利于 AI 助手在确认前做风险提示。

#### 7）如果用户挂号成功后要看什么？

可以看挂号记录、消息中心、后续电子处方、评价入口。

#### 8）当前支付链路是什么状态？

当前版本为了降低本地联调复杂度，支付链路做了简化，挂号单直接落到成功态，支付超时恢复等逻辑仍属于后续可继续增强方向。

---

### 8.3 视频问诊类

#### 9）视频问诊链路里最重要的设计点是什么？

把业务状态同步和音视频传输解耦：WebSocket 管业务状态，TRTC 管媒体流。

#### 10）医生端页面为什么复杂？

因为它既要维护上线/下线、开放/关闭、当前问诊、排队队列，还要响应 WebSocket 推送和 TRTC 房间状态，所以是典型的多状态协同页面。

#### 11）如果 WebSocket 掉线怎么办？

可以通过重连 + 主动 `refreshInfo` 拉取兜底，避免只依赖推送造成前端状态漂移。

---

### 8.4 电子处方类

#### 12）处方为什么要绑挂号单？

因为处方是一次就诊行为的后置产物，和医生、诊室、就诊卡、问诊记录天然相关，挂号单是最稳定的业务锚点。

#### 13）处方为什么用 JSON 存储药品列表？

列表结构灵活，前后端渲染方便，但要在保存时做格式校验，并承认它在统计分析上不如强结构化表。

#### 14）处方权限是怎么做的？

医生只能操作自己的挂号单，管理员或有特定权限的账号可查看全部，这在 `DoctorPrescriptionServiceImpl` 里做了显式权限判断。

---

### 8.5 AI 助手类

#### 15）AI 助手和普通聊天机器人的本质区别是什么？

普通聊天机器人偏问答，任务型 Agent 要能理解状态、调用工具、推进流程、控制写操作、输出可执行结果。

#### 16）为什么不能让大模型直接给挂号结果？

因为医疗挂号属于强约束业务，最终事实必须来自真实排班和号源数据，不能让模型靠语言概率猜。

#### 17）为什么还要保留规则引擎？

因为像“牙痛 -> 口腔科”“今天/明天/后天 -> 绝对日期”“最早/最晚 -> 时段偏好”这类模式化规则，规则引擎更稳定、可控、成本更低。

#### 18）AI 助手为什么要做确认弹窗？

因为挂号属于写操作，会产生真实业务影响，所以必须把读操作和写操作区分开，写操作统一走确认链路。

#### 19）会话记忆为什么放 Redis？

跨请求共享、方便横向扩展、可设置 TTL，且比只放前端更稳。

#### 20）如果 LLM 服务不可用怎么办？

可以降级到无模型规则引擎继续跑基础流程，保证助手至少还能做有限的任务推进，而不是整体不可用。

#### 21）做三套 Agent 架构有意义吗？

有。因为它能帮助比较不同模式下 LLM、工具和编排层的边界，最终沉淀出更适合强业务流程的工程方案。

#### 22）你们怎么评估 AI 助手好不好用？

可以看：

- 从发起到挂号成功的转化率
- 平均工具调用次数
- 进入确认态的比例
- 幻觉率 / 未验证回复率
- 用户在中途跳回原挂号页的比例
- 失败原因分布（无号、条件不满足、识别失败）

#### 23）如果用户一句话给了很多条件怎么办？

比如“帮我挂明天赵六最早的口腔科”，要先解析出科室、日期、医生和偏好，再调用工具验证真实候选，最后自动推进到确认态，而不是机械地重复问一遍。

#### 24）AI 助手怎么避免无限循环查工具？

要设置状态机和停点：何时可以自动推进，何时必须等用户确认，何时直接回退到更早查询步骤。

#### 25）你们最关键的 AI 工程结论是什么？

LLM 负责理解意图和选下一步动作，真实业务事实由工具产生，流程守卫由编排层承担，写操作必须确认。

## 9. Java 基础八股：别脱离项目，要会结合场景讲

这一节不建议死背定义，建议都尽量往当前项目上靠。

---

### 9.1 String、StringBuilder、StringBuffer 有什么区别？

- `String`：不可变对象，线程安全，但频繁拼接会产生较多中间对象
- `StringBuilder`：可变，线程不安全，单线程拼接推荐用它
- `StringBuffer`：可变，线程安全，性能相对低

**项目里怎么说**：

像消息文案、确认摘要、日志拼接这类场景，如果在局部方法里频繁组装字符串，优先 `StringBuilder`；如果只是简单模板拼接，直接 `String` 也够用。

---

### 9.2 HashMap 和 ConcurrentHashMap 有什么区别？

- `HashMap` 非线程安全
- `ConcurrentHashMap` 线程安全，适合并发读写

**项目里怎么说**：

WebSocket 会话池 `sessionMap` 用的是 `ConcurrentHashMap<String, Session>`，因为多个用户连接、断开、消息处理可能并发发生。

---

### 9.3 synchronized 和 ReentrantLock 区别？

- `synchronized`：JVM 层关键字，用法简单
- `ReentrantLock`：功能更丰富，可中断、可轮询、可公平锁

**项目里怎么说**：

如果我只是做对象级临界区保护，`synchronized` 就够；如果我要做更复杂的锁控制，比如可中断等待、超时获取锁，我会更倾向 `ReentrantLock`。不过当前挂号项目里并发控制主要放在 Redis 乐观锁层，而不是 JVM 本地锁。

---

### 9.4 volatile 能保证什么？

- 保证可见性
- 禁止指令重排
- **不能**保证复合操作原子性

**项目里怎么说**：

像“号源 +1”这种操作不能靠 `volatile`，因为它不是原子操作，所以项目里用的是 Redis 事务而不是本地变量控制。

---

### 9.5 ThreadLocal 有什么用？

可为每个线程存一份独立变量副本，常见于用户上下文、traceId 等。

**项目里怎么说**：

AI 助手的对话记忆没有用 ThreadLocal，因为它需要跨 HTTP 请求保存，ThreadLocal 只能活在线程生命周期里，所以最终选的是 Redis 会话记忆。

---

### 9.6 equals 和 hashCode 为什么要一起重写？

因为基于哈希的集合先看 `hashCode` 分桶，再通过 `equals` 判断对象是否相等；只重写一个会造成集合行为异常。

---

### 9.7 Java 内存模型你怎么理解？

核心关注点：

- 线程之间的可见性
- 原子性
- 有序性

**项目里怎么说**：

项目里真正有价值的不是背定义，而是知道：多线程共享状态不能想当然。比如 WebSocket 会话池需要并发安全容器，挂号库存不能靠本地变量控制，要么数据库锁，要么分布式缓存事务。

---

### 9.8 JVM 内存区域有哪些？

- 程序计数器
- 虚拟机栈
- 本地方法栈
- 堆
- 方法区 / 元空间

#### 面试怎么结合项目讲

- WebSocket 长连接、Redis 缓存对象、页面大列表都可能带来堆内存压力
- AI 助手如果把工具结果、会话上下文堆积太多，也会推高内存占用，所以需要 TTL、裁剪和必要的状态暴露

---

### 9.9 常见 GC 算法和垃圾收集器？

- 标记清除
- 复制算法
- 标记整理
- 分代收集

常见收集器：Serial、ParNew、CMS、G1。

**项目里怎么讲**：

如果线上出现 Full GC 频繁，优先排查是不是：

- WebSocket 会话没释放
- 大量 JSON / 列表对象堆积
- Redis 返回结果没有必要缓存过久
- AI 会话 memory 过大或保留时间过长

## 10. Spring / Spring Boot 八股

---

### 10.1 IOC 和 AOP 分别是什么？

- IOC：对象创建和依赖关系交给 Spring 容器管理
- AOP：把横切逻辑（日志、事务、鉴权）统一织入

**项目里怎么说**：

像 `@Service`、`@Resource` 就是 IOC 在管理业务组件；事务、权限校验、日志都适合用 AOP 思路处理。

---

### 10.2 Spring 事务什么时候会失效？

常见情况：

- 同类方法内自调用
- 方法不是 `public`
- 抛出的异常类型没命中回滚规则
- 代理没生效

**项目里怎么结合当前项目说**：

挂号这种“缓存 + DB + 消息”的链路，不能误以为一个 `@Transactional` 就全包了，因为 Redis、WebSocket、外部支付都不在数据库本地事务里，所以要用补偿思维而不是只靠声明式事务。

---

### 10.3 Bean 生命周期大概流程？

实例化 -> 属性注入 -> 初始化回调 -> 可用 -> 销毁。

---

### 10.4 Spring Boot 自动配置原理？

核心是：

- `@SpringBootApplication`
- 自动配置类
- 条件装配 `@Conditional`
- 从配置文件绑定属性

**项目里怎么说**：

当前项目大量依赖配置驱动，比如数据库、Redis、Sa-Token、Quartz、Agent 参数等，都体现了 Spring Boot 约定优于配置和条件装配的思想。

---

### 10.5 为什么项目里会用 `@EnableAsync`？

表示开启异步任务能力，适合处理不阻塞主流程的业务，比如部分通知类任务、异步线程任务等。

---

### 10.6 `@MapperScan` 的作用？

让 MyBatis Mapper 接口被 Spring 扫描并注册为 Bean。

**项目里落点**：

- `HospitalApiApplication.java`
- `PatientWxApiApplication.java`

## 11. MyBatis 八股

---

### 11.1 MyBatis 和 JPA 有什么区别？

- MyBatis 更偏 SQL 显式控制
- JPA 更偏对象关系映射和约定式开发

**项目里怎么说**：

这个项目业务 SQL 比较多，像挂号、排班、评价、收藏、处方、消息这些查询条件较复杂，所以使用 MyBatis 会更便于精准控制 SQL。

---

### 11.2 #{} 和 ${} 区别？

- `#{}`：预编译参数，占位安全，推荐使用
- `${}`：字符串拼接，存在 SQL 注入风险

**项目里怎么说**：

医疗项目涉及真实用户数据，动态 SQL 一定要优先用 `#{}`，避免拼接带来的注入风险。

---

### 11.3 MyBatis 一级缓存和二级缓存？

- 一级缓存：SqlSession 级别，默认开启
- 二级缓存：Mapper 级别，需要显式配置

**项目里怎么说**：

像挂号、时段、消息这种实时性较强的数据，我会谨慎使用缓存，否则容易读到过期状态。

---

## 12. MySQL 八股

---

### 12.1 InnoDB 和 MyISAM 区别？

- InnoDB 支持事务、行锁、外键
- MyISAM 不支持事务，偏读多场景

**项目里怎么说**：

挂号、消息、评价、处方都是强事务业务，当然优先 InnoDB。

---

### 12.2 索引为什么能加快查询？

因为索引减少了全表扫描，常见底层结构是 B+ 树。

**项目里怎么说**：

像挂号记录按用户、按日期、按医生、按诊室查询都很常见，这些高频过滤字段应该重点考虑索引设计。

---

### 12.3 事务四大特性 ACID？

- 原子性
- 一致性
- 隔离性
- 持久性

**项目里怎么说**：

挂号写入、处方保存、评价提交这类业务都依赖事务，但同时也要承认：跨 Redis、消息、支付这类链路不能只靠数据库事务，要做更完整的一致性设计。

---

### 12.4 事务隔离级别有哪些？

- 读未提交
- 读已提交
- 可重复读
- 串行化

#### 幻读、不可重复读、脏读要会区分

**项目里怎么说**：

如果只靠数据库控制挂号库存，高并发下就会涉及锁竞争与隔离级别问题，这也是项目最终选择把号源抢占前置到 Redis 的原因之一。

---

### 12.5 什么是最左前缀原则？

联合索引从最左列开始匹配时才更容易命中索引。

**项目里怎么说**：

像“按用户 + 日期 + 状态”查挂号记录、“按医生 + 时间范围”查排班，这些都适合从查询最常用组合出发设计联合索引。

---

## 13. Redis 八股

---

### 13.1 Redis 为什么快？

- 基于内存
- 单线程模型减少上下文切换（现代版本还有多线程 IO 优化）
- 数据结构高效
- 网络开销小

**项目里怎么说**：

挂号时段库存、AI 对话 memory 都要求低延迟访问，所以非常适合放 Redis。

---

### 13.2 Redis 常见数据类型有哪些？

- String
- Hash
- List
- Set
- ZSet

**项目里怎么说**：

- 会话记忆可以直接存对象序列化 / JSON
- 时段号源更适合 Hash，能维护 `maximum` 和 `num`

---

### 13.3 缓存穿透、击穿、雪崩分别是什么？

- 穿透：查不存在的数据，缓存和数据库都被打穿
- 击穿：热点 key 过期，大量请求同时打到 DB
- 雪崩：大量 key 同时过期

**项目里怎么说**：

挂号时段 key、AI session key 都需要合理 TTL 和失效策略，尤其是 AI memory 如果 TTL 太短，用户会频繁丢上下文；TTL 太长，又会占用更多 Redis 空间。

---

### 13.4 Redis 持久化方式？

- RDB
- AOF

**项目里怎么说**：

像 AI memory 这种状态丢了还能恢复部分流程，容忍度比库存类缓存高；而号源控制则更关注实时一致性和过期管理。

---

### 13.5 为什么用 Redis 保存 Agent 会话，而不是数据库？

- 读写频繁、状态短期有效
- TTL 明确
- 不需要长期历史留档
- 对多轮对话更友好

---

## 14. WebSocket / 网络通信八股

---

### 14.1 WebSocket 和 HTTP 有什么区别？

- HTTP 主要是请求-响应
- WebSocket 是长连接，服务端可主动推送

**项目里怎么说**：

视频问诊状态、医生端实时刷新这类场景更适合 WebSocket；普通查询接口仍然用 HTTP。

---

### 14.2 为什么消息推送不用轮询？

轮询实时性差且浪费资源，WebSocket 在需要实时推送时更合适。

### 14.3 WebSocket 断线重连要注意什么？

- 身份重建
- 状态补偿
- 防止重复连接导致旧连接未清理

**项目里怎么说**：

当前后端用 `ConcurrentHashMap<String, Session>` 管理连接，会根据用户身份映射会话；前端断线重连后最好结合一次状态刷新，避免仅靠推送恢复。

## 15. Quartz 八股

---

### 15.1 为什么用 Quartz？

因为就诊提醒属于后端任务调度，适合由服务端统一执行，不依赖前端在线。

### 15.2 Quartz 核心概念有哪些？

- Job
- Trigger
- Scheduler

**项目里怎么说**：

- `RegistrationReminderJob` 是 Job
- `registrationReminderTrigger()` 定义了 Trigger
- Spring Boot 集成 Quartz 负责调度

### 15.3 多实例部署下 Quartz 会有什么问题？

可能重复执行，所以需要结合持久化 JobStore、分布式锁或幂等机制设计。

## 17. AI 应用开发专题：这部分要讲出“工程味”

这部分是本面经最值得准备的章节。

---

### 17.1 什么叫 AI 应用开发，不只是调模型 API

AI 应用开发至少要回答下面几个问题：

1. 模型负责什么，不负责什么？
2. 工具边界怎么定义？
3. 写操作怎么保护？
4. 多轮状态怎么保存？
5. 出错怎么降级？
6. 怎么评估效果？
7. 如何防止幻觉和越权？

如果面试官问“你做过 AI 应用开发吗”，你可以直接围绕这 7 点展开。

---

### 17.2 为什么任务型 Agent 比普通聊天更难

因为任务型 Agent 要同时处理：

- 自由文本理解
- 业务规则校验
- 多轮状态推进
- 真实工具调用
- 异常分支与降级
- 写操作确认

**项目里的真实体现**：

- 用户可以说“帮我挂明天口腔科的号”
- 也可以说“赵六最早的”
- 还可以点卡片、改日期、换医生、确认挂号

系统不能只会“回答”，而要真正“推进任务”。

---

### 17.3 当前 AI 助手的工具边界怎么设计

#### 只读工具

- 查科室
- 查诊室
- 查日期
- 查医生
- 查时段
- 查消息
- 查就诊卡状态

#### 需确认工具

- 校验挂号条件
- 创建挂号单

#### 面试时这样说

> 我们把工具分成两类：只读工具和写工具。只读工具自动执行，用来查事实；写工具必须确认，用来真正改变业务状态。这样能让 Agent 更像一个可信执行器，而不是随时可能误操作的聊天机器人。

---

### 17.4 如何降低幻觉

#### 当前项目采用的方式

1. 工具优先，事实来自工具
2. 用户原话优先，不让模型强行覆盖关键字段
3. 单候选自动推进，多候选明确停点
4. 写操作前二次校验
5. 需要时回退到更早步骤，而不是硬编结果
6. 模型不可用时可降级到规则引擎

#### 面试回答模板

> 降低幻觉不能只靠 prompt，而要靠系统设计。我的做法是让模型尽量少碰最终事实，把事实生成权收回到工具层，把业务约束收回到编排层。模型做理解，工具做查询，编排层做决策守卫。

---

### 17.5 为什么一期保留规则引擎，二期三期再做对照

因为：

- 基础业务规则明确，规则引擎成本更低、确定性更强
- 某些语义理解交给模型更灵活
- 通过三套架构对照，能找到更适合医疗挂号场景的方案

这是一种非常典型的 AI 工程思路：

> 不迷信全模型驱动，而是按业务风险把“规则、模型、编排、工具”合理分层。

---

### 17.6 AI 助手的会话记忆应该存什么，不该存什么

#### 应该存

- 当前阶段 `stage`
- 科室 / 诊室 / 日期 / 医生 / 时段
- 待确认订单 `pendingOrder`
- 上一轮 observation 的核心候选

#### 不应该无脑全存

- 全量工具原始返回
- 过大的历史日志
- 过期上下文
- 与当前任务无关的噪音信息

#### 为什么

因为 memory 不是聊天记录归档系统，而是任务推进所需的最小业务状态。

---

### 17.7 AI 助手怎么评估效果

这是很多人最容易忽略的点。

你可以从 4 个维度答：

#### 1）任务完成率

- 从发起到进入确认态的比例
- 从发起到挂号成功的比例

#### 2）过程效率

- 平均对话轮数
- 平均工具调用次数
- 单候选自动推进比例

#### 3）质量指标

- 幻觉率
- 错误实体识别率
- 用户回退到手动页面的比例

#### 4）稳定性指标

- LLM 超时率
- 降级触发率
- 工具错误分布

这四类一讲出来，面试官基本就知道你不是“Demo 式 AI”。

---

### 17.8 如何设计 AI 应用的降级策略

可以答：

- LLM 不可用：退到无模型规则引擎
- 工具失败：明确告诉用户哪个环节失败，不编结果
- 写操作前校验失败：停在确认前，不落单
- 候选不足：引导用户返回原挂号页面

一句话总结：

> 降级不是“能回答点什么就行”，而是“即使部分能力失效，也不能给出错误业务结果”。

---

### 17.9 AI 应用开发面试的压轴回答

如果面试官问：**“你觉得做 AI 应用最重要的是什么？”**

你可以这样说：

> 我觉得最重要的不是模型本身，而是边界设计。要先定义模型负责理解，工具负责事实，编排层负责状态和安全，最后再考虑 prompt 和交互优化。尤其在医疗挂号这种强业务约束场景里，如果边界不清晰，模型越智能，风险反而越大。

这个回答非常适合压轴。

## 18. 场景设计题：面试官很爱问

---

### 场景 1：如果挂号并发量突然翻 10 倍，你怎么优化？

可以答：

1. 把并发竞争前置到 Redis
2. 对热点时段做缓存预热
3. 挂号写库走异步化/批量化要慎重，因为医疗业务更看重一致性
4. 增加失败重试与对账补偿
5. 对热门医生 / 热门时段做限流保护

---

### 场景 2：如果让你把 AI 助手做成生产级，你会补什么？

可以答：

1. 增加统一 traceId 和 tool log 观测
2. 增加离线评估集和回放测试
3. 完善 prompt / memory / observation 裁剪策略
4. 加强写操作幂等和权限保护
5. 做指标看板：成功率、降级率、失败原因分布

---

### 场景 3：如果让你从三套 Agent 中选一套继续演进，你怎么选？

可以答：

> 我会优先保留一期规则编排的稳定性，同时吸收二期和三期在自由文本理解和偏好解析上的优点。因为在当前业务里，“稳定正确”比“链路看起来更智能”更重要。

---

### 场景 4：如果用户投诉 AI 助手“总是重复问问题”，你怎么优化？

可以答：

1. 增加单候选自动推进
2. 命中用户已有条件时直接下钻
3. 记住前序上下文，避免重复收集
4. 优化停点逻辑，只在真正需要用户选择时停下

---

### 场景 5：如果视频问诊中 WebSocket 推送丢了怎么办？

可以答：

1. 推送只是加速通道，不是唯一事实来源
2. 页面要有主动刷新接口兜底
3. 关键状态以服务端真实状态为准
4. 重连后主动做一次状态同步

# 智慧医疗预约挂号系统简历素材

## 项目概述

- 项目名称：智慧医疗预约挂号系统
- 技术栈：`Spring Boot 2.7`、`MyBatis`、`MySQL`、`Redis`、`Sa-Token`、`Quartz`、`WebSocket`、`Vue3`、`Element Plus`、`UniApp`、`TRTC`
- 项目定位：面向患者、医生和管理员的一体化医疗服务平台，包含患者小程序、后台管理系统及双后端服务，支持在线挂号、医生排班、电子处方、消息提醒、医生收藏、视频问诊和 AI 挂号助手等功能。

## STAR 项目经历

### 版本一：适合校招简历

**项目名称：智慧医疗预约挂号系统**

**项目描述：**

面向患者、医生和管理员的一体化医疗服务平台，包含患者小程序、后台管理系统及双后端服务，支持在线挂号、医生排班、电子处方、消息提醒、医生收藏、视频问诊和 AI 挂号助手等功能。

**个人职责：**

- 负责患者端预约挂号主流程的接口开发，完成科室、日期、医生、时段查询以及挂号条件校验、挂号提交等核心能力建设。
- 基于 Redis 事务机制实现号源并发控制，避免高并发场景下重复挂号和号源超卖。
- 参与开发电子处方模块，打通 MIS 端医生开方与患者端按挂号单查询处方详情的业务闭环。
- 参与实现消息中心与就诊提醒功能，使用 Quartz 定时推送每日就诊提醒，并结合站内消息提升患者触达能力。
- 参与小程序 AI 挂号助手能力建设，通过规则引擎、会话记忆和意图识别实现对话式挂号引导。

**项目亮点：**

- 完成患者端“登录/就诊卡/挂号/消息/收藏/评价”业务闭环。
- 后端采用 `Spring Boot + MyBatis + Redis + Sa-Token`，前端采用 `Vue3 + Element Plus + UniApp`。
- 项目涉及权限控制、定时任务、实时通知、音视频问诊等典型业务场景，具备较强的综合实践价值。

### 版本二：适合社招简历

**项目名称：智慧医疗预约挂号系统**

**项目描述：**

负责互联网医疗系统核心业务模块开发，系统包含管理端 MIS、患者端小程序和后端服务，支持挂号预约、电子处方、消息通知、收藏医生、视频问诊及 AI 挂号辅助等能力。

**核心贡献：**

- 主导预约挂号核心链路开发，设计并实现科室、医生、时段查询、挂号条件校验、挂号落单等接口，保障患者侧挂号流程完整可用。
- 在挂号链路中引入 Redis `watch/multi/exec` 控制号源并发，降低热门时段下的超卖风险，并同步更新挂号记录、排班余量和通知消息。
- 负责电子处方模块的后端实现，完成挂号单关联、处方 JSON 明细校验、医生本人权限隔离及患者端处方查询闭环。
- 推动患者端消息体系建设，落地未读消息统计、消息已读、Quartz 每日提醒任务，并通过 WebSocket 支持实时通知能力。
- 参与 AI 挂号助手一期建设，设计对话式挂号编排逻辑，引入 Redis 会话记忆和确认机制，实现从自然语言输入到挂号确认的链路打通。

**结果产出：**

- 打通患者从挂号到处方查看的完整业务闭环。
- 提升挂号操作效率，减少用户多页面跳转成本。
- 增强系统在高并发预约、权限隔离、消息提醒和智能引导等方面的工程完整度。

### 版本三：STAR 拆解版

#### 1. 挂号核心链路

- S：项目需要同时支持患者端在线挂号、排班查询和号源管理。
- T：负责预约挂号主流程设计与接口打通。
- A：开发科室、日期、医生、时段查询、挂号条件校验、挂号提交等接口，并在挂号提交中基于 Redis `watch/multi/exec` 做号源并发控制，同步落库挂号记录和站内消息提醒。
- R：打通患者从选科室到提交挂号的完整闭环，降低高并发下号源超卖风险。

#### 2. AI 挂号助手

- S：传统挂号路径步骤较多，患者需要反复查询科室、医生和号源。
- T：在小程序侧新增自然语言挂号入口。
- A：落地 `/agent/chat` 接口，基于规则引擎、DashScope 意图识别和 Redis 会话记忆，实现症状映射科室、分步引导挂号、挂号前确认、失败回退原流程等能力。
- R：把原本多页面跳转的挂号流程收敛成对话式引导，提升挂号入口的可用性和产品亮点。

#### 3. 电子处方闭环

- S：医生开方和患者查看处方缺少统一闭环。
- T：负责 MIS 端处方管理与患者端处方查询能力。
- A：开发处方分页查询、按挂号单查询、处方保存和编辑接口，增加医生本人数据隔离和权限校验，对 JSON 药品明细做解析与校验，并在患者端开放按挂号单查看电子处方详情。
- R：打通“挂号单 -> 医生开方 -> 患者查看”的完整链路，提升医生侧录入效率和处方数据一致性。

#### 4. 消息提醒与用户运营能力

- S：项目需要提升患者复诊效率和就诊提醒能力。
- T：完善消息中心、收藏医生、个人中心聚合能力。
- A：开发未读消息查询、消息已读、每日就诊提醒 Quartz 定时任务，支持患者收藏和取消收藏医生、个人中心收藏预览、后台收藏管理，并通过 WebSocket 支持实时通知。
- R：补齐患者消息触达和常用医生复用场景，增强系统留存和复诊体验。

## 1 分钟自我介绍话术

我最近做的项目是一个智慧医疗预约挂号系统，整体包括后台管理端、患者小程序和后端服务。项目技术栈主要是 `Spring Boot`、`MyBatis`、`MySQL`、`Redis`、`Sa-Token`，前端用了 `Vue3`、`Element Plus` 和 `UniApp`。

我主要参与的是后端核心业务开发，重点做了三个方向。第一是预约挂号主链路，包括科室、医生、排班、时段查询和挂号提交，其中我用 Redis 事务控制号源并发，避免超卖。第二是电子处方模块，打通了医生在管理端开方、患者在小程序查看处方的完整流程，同时补了权限校验和数据校验。第三是 AI 挂号助手一期，通过规则编排、会话记忆和意图识别，把原来多页面操作改成对话式挂号引导。

这个项目让我对医疗类业务中的权限控制、并发处理、定时任务、消息通知和业务闭环设计有了比较完整的实践经验。

## 面试可能追问的问题

- 你为什么用 Redis 事务控制号源，而不是只靠数据库锁？
- 挂号提交流程里，如何保证 Redis 和 MySQL 数据一致性？
- 电子处方为什么要做医生本人权限隔离？具体怎么做？
- Quartz 定时提醒如果重复执行，怎么避免重复发消息？
- AI 挂号助手和普通搜索页相比，价值在哪里？
- 如果号源很紧张，系统怎么做限流、兜底和失败提示？

## 建议补的知识点

### 第一优先级

- `Spring Boot`：参数校验、统一异常处理、AOP、事务传播、接口分层设计。
- `MyBatis + MySQL`：动态 SQL、索引设计、分页查询、执行计划、事务隔离级别、乐观锁和悲观锁。
- `Redis`：缓存设计、会话存储、`watch/multi/exec` 事务、分布式锁、缓存一致性。
- `Sa-Token / RBAC`：登录态、权限点、菜单权限、数据权限隔离。

### 第二优先级

- `Quartz`：Cron 表达式、定时任务幂等、失败重试、任务监控。
- `WebSocket`：心跳、断线重连、在线状态维护、实时通知设计。
- `UniApp / 微信小程序`：登录授权、页面通信、请求封装、性能优化。
- `Vue3 + Element Plus`：组件封装、路由守卫、权限菜单、后台管理系统常见模式。

### 第三优先级

- `TRTC / WebRTC 基础`：房间模型、信令流程、音视频接入流程。
- `AI Agent`：意图识别、状态机编排、tool calling、会话记忆、确认机制、Prompt 设计。
- `工程化`：单元测试、接口测试、日志追踪、Docker 部署、Nginx 反向代理。

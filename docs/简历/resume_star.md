**2025.12～至今**

**智能医疗预约挂号系统**

**项目角色：项目负责人**

**项目背景：**

面向医院场景的智能挂号系统，涵盖高并发库存管理、AI 对话式挂号、智能排队叫号三大核心模块，解决传统挂号流程繁琐、高并发超卖、患者排队体验差等痛点。开源地址：<https://gitee.com/LtAo857/hospital>

**技术选型：**

`Spring Boot 3.1.5`、`MyBatis-Plus`、`Redis`、`MySQL`、`RabbitMQ`、`Vue 3`、`UniApp`、`Claude AI`

**工作内容：**

1. 设计开发注解驱动的 AI 工具系统，基于 `@AITool` 和 `@AIParam` 实现工具方法声明式定义，开发 `AIToolRegistry` 工具注册器并通过反射自动扫描注册工具方法；实现参数自动类型转换和必填校验，新增工具仅需补充注解方法无需修改核心代码，开发效率提升 80%，工具调用成功率提升至 95%。
2. 解决 AI 工具调用死循环及上下文丢失问题，通过实现工具调用次数限制和 30 秒超时控制，使用 Redis 存储对话历史与工具调用计数；将原有 100+ 行复杂 Prompt 精简为 15 行核心规则，设计标准化工具调用格式 `TOOL_CALL: tool_name({"param":"value"})`，使 API 成本降低 70%，平均 3-5 轮对话即可完成挂号。
3. 构建 AI 防御性编程与状态管理体系，通过参数自动类型转换和必填校验降低工具调用出错率，工具执行结果自动回传，确保 AI 基于真实数据回复；使用 `CompletableFuture` 实现超时兜底，用户等待超过 30 秒自动返回友好提示；实现会话过期自动清理和断点续传机制，支持对话中断后继续未完成流程。
4. 解决高并发抢号场景下的库存超卖问题，通过 Redis + Lua 脚本实现原子化库存扣减，结合 RabbitMQ 异步订单处理，将系统 QPS 提升至 1000+，响应时间控制在 50ms 以内，库存准确率达到 100%，彻底杜绝超卖。
5. 优化系统存储与并发架构，设计 Redis 缓存 + 定时同步机制，库存操作在 Redis 中同步完成，订单写入 MySQL 异步处理，通过手动 ACK 和失败重试保障最终一致性，大幅降低数据库压力。

### 版本零-A：AI 强化版

**2025.12～至今**

**智能医疗预约挂号系统**

**项目角色：项目负责人**

**项目背景：**

面向医院场景的智能挂号系统，重点建设 AI 对话式挂号能力，解决传统挂号流程步骤多、用户需要反复筛选科室医生、操作路径长的问题。

**技术选型：**

`Spring Boot`、`MyBatis`、`Redis`、`MySQL`、`RabbitMQ`、`UniApp`、`Claude AI`

**工作内容：**

1. 主导 AI 对话式挂号模块设计与实现，将挂号流程抽象为多轮对话状态机，围绕“信息补全 -> 候选筛选 -> 挂号确认”构建面向业务执行的交互链路。
2. 设计并对比三套 Agent 架构：第一套采用规则编排 + LLM 弱辅助模式，强调状态机控制和业务稳定性；第二套采用传统 ReAct 工具循环模式，验证 LLM 驱动工具调用的可行性；第三套采用 CC 风格计划执行模式，探索计划式执行在挂号场景下的可解释性与可扩展性。
3. 基于 Redis 实现会话记忆与状态推进，缓存用户在当前会话中的关键上下文和确认状态，支持多轮连续对话，并减少用户重复输入。
4. 针对大模型在业务场景中容易出现的日期误判、流程回退、提前结束、上下文丢失和误触发写操作等问题，增加阶段守卫、工具回传、候选自动推进、写操作确认、超时兜底与异常回退机制，提升系统稳定性与可控性。
5. 结合预约挂号核心链路与高并发库存控制能力，为 AI 挂号助手提供统一业务支撑，使用户可以通过 3-5 轮对话完成挂号核心流程，减少页面跳转成本并提升使用体验。

## 1 分钟自我介绍话术

我最近做的项目是一个智能医疗预约挂号系统，整体包括后台管理端、患者小程序和后端服务。项目技术栈主要是 `Spring Boot`、`MyBatis`、`MySQL`、`Redis`、`Sa-Token`，前端用了 `Vue3`、`Element Plus` 和 `UniApp`。

我主要参与的是后端核心业务开发，重点做了三个方向。第一是预约挂号主链路，包括科室、医生、排班、时段查询和挂号提交，其中我用 Redis 事务控制号源并发，避免超卖。第二是电子处方模块，打通了医生在管理端开方、患者在小程序查看处方的完整流程，同时补了权限校验和数据校验。第三是 AI 挂号助手一期，通过规则编排、会话记忆和意图识别，把原来多页面操作改成对话式挂号引导。

这个项目让我对医疗类业务中的权限控制、并发处理、定时任务、消息通知和业务闭环设计有了比较完整的实践经验。

- 电子处方为什么要做医生本人权限隔离？具体怎么做？

## 建议补的知识点

### 第一优先级

- `Spring Boot`：参数校验、统一异常处理、AOP、事务传播、接口分层设计。
- `MyBatis + MySQL`：动态 SQL、索引设计、分页查询、执行计划、事务隔离级别、乐观锁和悲观锁。
- `Redis`：缓存设计、会话存储、`watch/multi/exec` 事务、分布式锁、缓存一致性。
- `Sa-Token / RBAC`：登录态、权限点、菜单权限、数据权限隔离。

### 第二优先级

- `Quartz`：Cron 表达式、定时任务幂等、失败重试、任务监控。
- `WebSocket`：心跳、断线重连、在线状态维护、实时通知设计。
- `UniApp / 微信小程序`：登录授权、页面通信、请求封装、性能优化。
- `Vue3 + Element Plus`：组件封装、路由守卫、权限菜单、后台管理系统常见模式。

### 第三优先级

- `TRTC / WebRTC 基础`：房间模型、信令流程、音视频接入流程。
- `AI Agent`：意图识别、状态机编排、tool calling、会话记忆、确认机制、Prompt 设计。
- `工程化`：单元测试、接口测试、日志追踪、Docker 部署、Nginx 反向代理。
